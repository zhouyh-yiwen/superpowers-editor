# Receiving Review

Use this skill when review comments already exist and the next job is to process them rigorously. The goal is not to praise the reviewer. The goal is to understand the feedback, decide what is valid, and update the work product one issue cluster at a time.

**Announce at start:** "I'm using the receiving-review skill to process review feedback before revising."

## Use This For

- Handling AI critique of a proposal plan
- Processing review comments on a writing brief
- Turning a list of objections into a revision sequence
- Deciding which review points to accept, question, or reject

## Do Not Use This For

- Requesting a fresh review
- Brainstorming a new direction before reviewing current feedback
- Blindly applying every comment without checking relevance

If no feedback exists yet, use `requesting-review`.

## Core Rules

- Read all feedback before changing anything.
- Separate understanding from agreement.
- Accept, question, or reject each point explicitly.
- Merge overlapping short comments into a few revision clusters.
- Revise one issue cluster at a time.
- Preserve the artifact's main purpose while fixing weak spots.
- Push back when feedback weakens the artifact or misunderstands the goal.
- Distinguish repeated substantive concerns from one-off stylistic preferences.

## Process

1. **Read the full feedback**
   Do not start editing after the first comment.

2. **Restate the points**
   Put each item in your own words so the next step is clear.

3. **Classify each point**
   For each comment, decide:
   - accept
   - question
   - reject

4. **Group revisions**
   Cluster related comments into revision groups such as:
   - audience clarity
   - title or framing
   - selling point clarity
    - weak proof
    - repetition
    - missing market logic
   - comparables / distinctness
   - over-breadth / focus drift
   - saleability / spreadability
    - marketing route
    - angle drift
    - tone or route confusion

5. **Revise deliberately**
   Update the artifact one group at a time.

6. **Summarize the result**
   State what changed, what remains open, and what was intentionally not changed.

## Output Format

### Review Summary
- short restatement of what the feedback is saying overall

### Decision Table
- item
- role or lens
- accept / question / reject
- reason

### Revision Groups
- grouped issues that should be fixed together

### Revisions To Make
1. ...
2. ...
3. ...

### Open Points
- any comments that still need clarification or a deliberate decision

## Quality Bar

Good review handling:
- distinguishes valid critique from irrelevant critique
- protects the artifact's purpose
- makes revision order visible
- explains why some comments are not adopted
- consolidates repeated comments into a smaller number of real edits

Bad review handling:
- blindly accepts everything
- argues with everything
- mixes unrelated fixes together
- revises before understanding the full feedback
- treats each short comment as a fully separate rewrite task when several are saying the same thing
