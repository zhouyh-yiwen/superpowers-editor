# Using Editorial Superpowers

These skills are not a full software-delivery workflow anymore. They are editorial decision tools for key publishing moments.

## The Rule

Before responding, decide whether the task is mainly:

- **still fuzzy** and needs angle finding, audience judgment, or option comparison  
  → use `brainstorming`

- **mostly decided** and needs structure, order, or a report plan  
  → use `proposal-planning`

- **angle chosen** and needs a reader-facing writing route  
  → use `copywriting-brief`

- **artifact ready** and needs independent critique  
  → use `requesting-review`

- **review already received** and needs revision handling  
  → use `receiving-review`

If both are needed, use them in order:

- `brainstorming` → `proposal-planning` for internal reports
- `brainstorming` → `copywriting-brief` for public-facing writing
- `proposal-planning` → `requesting-review` before a topic meeting
- `copywriting-brief` → `requesting-review` before drafting
- `requesting-review` → `receiving-review` after feedback arrives

## Quick Guide

Use `brainstorming` for:
- book positioning
- audience exploration
- selling-point exploration
- post angle exploration
- recommendation hook comparison
- extracting candidate angles and usable materials from a full manuscript or EPUB

Use `proposal-planning` for:
- proposal report structure
- writing order
- section purpose
- evidence gaps
- report planning

Use `copywriting-brief` for:
- article brief
- recommendation post route
- editorial guidance
- article guidance
- writing route
- title direction
- opening hook
- middle-point planning
- ending landing point

Use `requesting-review` for:
- ask an AI reviewer to critique a proposal plan
- ask an AI reviewer to challenge a writing brief
- stress-test an artifact before sharing or revising

Use `receiving-review` for:
- process AI review comments
- decide what to accept, question, or reject
- turn feedback into a revision sequence

## Working Principle

Instructions say **what** the editor needs.  
The skills help determine **how** to think and organize the work.

## Output Discipline

- Do not force a full workflow when a single skill is enough.
- Do not generate final marketing copy when the task is only to explore angles.
- Do not produce a vague outline when the task requires a usable proposal plan.
- Do not force a full workflow when the user only needs one node.
- Do not create a separate reading workflow when the real task is editorial judgment from source material.
- Do not request review when feedback already exists.
- Do not process review comments before reading them all.
